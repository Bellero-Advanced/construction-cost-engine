\section{ภาพรวมของระเบียบวิธีวิจัย}
\label{sec:research-method}

งานวิจัยนี้ใช้ระเบียบวิธีวิจัยและพัฒนา (Research and Development: R\&D)
ผสมผสานกับการวิจัยเชิงปริมาณ (Quantitative Research) โดยแบ่งกระบวนการ
ออกเป็น 5 ระยะตามแนวทางวิศวกรรมซอฟต์แวร์ คือ
(1) ระยะวิเคราะห์ความต้องการ
(2) ระยะออกแบบสถาปัตยกรรม
(3) ระยะพัฒนา
(4) ระยะทดสอบ
(5) ระยะประเมินผลและปรับปรุง.

\subsection{สถาปัตยกรรมระบบ}

ระบบที่พัฒนาขึ้นใช้สถาปัตยกรรมแบบ \emph{edge-first serverless} โดยรัน
บน Cloudflare Workers ซึ่งเป็นแพลตฟอร์มที่กระจาย server ไปยัง
data center กว่า 300 แห่งทั่วโลก \cite{cloudflare2025}. การออกแบบนี้
ส่งผลให้ระบบมี \emph{cold-start latency เป็นศูนย์} และสามารถตอบ
request ได้ในหลักมิลลิวินาที (ค่าเฉลี่ย < 50 ms) จากทุกที่ในโลก
ซึ่งเหมาะสมกับงานวิจัยที่ต้องการให้ผู้ประมาณราคาเข้าถึงข้อมูลได้
ตลอดเวลา.

\section{การออกแบบฐานข้อมูล}
\label{sec:db-design}

ระบบใช้ \emph{Cloudflare KV (Key-Value Store)} แทน relational database
แบบดั้งเดิม เนื่องจากการเข้าถึงข้อมูลหลักของระบบเป็นการ lookup ด้วย
ชุดของ \texttt{(source, materialId, province)} ไม่จำเป็นต้องทำการ
JOIN ข้อมูลที่ซับซ้อน. KV ตอบสนองในระดับ 5--20 ms ทั่วโลก ซึ่งเร็วกว่า
PostgreSQL หรือ MySQL ในงาน lookup ลักษณะนี้.

\subsection{โครงสร้าง Key-Value}

ระบบจัดเก็บข้อมูล 3 ประเภทหลักใน KV ดังตารางที่ \ref{tab:kv-schema}.

\begin{table}[H]
\centering
\caption{โครงสร้าง KV Keys ที่ใช้ในระบบ}
\label{tab:kv-schema}
\begin{tabularx}{\textwidth}{l X X r}
\toprule
\textbf{ประเภท} & \textbf{Key Pattern} & \textbf{Value} & \textbf{TTL} \\
\midrule
Live Cache    & \texttt{\{source\}:\{materialId\}:\{province\}}              & \{price, fetchedAt\}            & 14--30 d \\
History       & \texttt{history:\{source\}:\{materialId\}:\{province\}}      & [\{date, price\}, \ldots]       & 365 d \\

\textbf{Live Cache} เก็บราคาปัจจุบันที่ดึงล่าสุดจากแต่ละแหล่ง โดยมี
อายุการเก็บ (TTL) ตั้งแต่ 14 ถึง 30 วันตามประเภทแหล่ง — แหล่งภาครัฐ
ที่อัปเดตช้า (CGD รายไตรมาส) ใช้ TTL ยาวกว่า, แหล่ง retail ที่ราคา
เปลี่ยนเร็ว (HomePro, MegaHome) ใช้ TTL สั้นกว่า.

\textbf{History} เก็บ time-series ของราคา เพื่อใช้ในการวิเคราะห์
แนวโน้ม โดยมี cron job เก็บ snapshot รายสัปดาห์ทุกวันจันทร์ 03:17 UTC.

\textbf{Calculations} เก็บผลการคำนวณ BOQ ของผู้ใช้ทุกครั้ง เพื่อใช้
เป็น dimension table \texttt{fact\_calculations} สำหรับการวิเคราะห์
เชิงรวม (aggregate analytics) ในอนาคต.

\section{การดึงและจัดการข้อมูล}
\label{sec:data-acquire}

\subsection{สี่กลยุทธ์การดึงข้อมูลราคา}

ระบบใช้ 4 กลยุทธ์ที่แตกต่างกันเพื่อรองรับลักษณะของแต่ละแหล่งข้อมูล
ดังนี้

\subsubsection{กลยุทธ์ที่ 1: PDF Parsing}

ใช้กับ TPSO และ CGD ที่เผยแพร่ข้อมูลในรูปแบบ PDF rate report.
ระบบใช้ library \texttt{unpdf} ซึ่งเป็น WASM-based PDF parser
ที่รัน Cloudflare Workers ได้. ขั้นตอน:

\begin{enumerate}[leftmargin=2em,itemsep=0.2em]
  \item Download PDF จาก URL ของแหล่ง
  \item Extract ข้อความจากทุกหน้าด้วย \texttt{unpdf.extractText()}
  \item ใช้ Regular Expression (RegEx) จับคู่บรรทัดที่ตรงกับ
        material code และ unit price
  \item Validate ราคาที่จับได้ (ต้องเป็นตัวเลขบวก, อยู่ในช่วงที่
        สมเหตุสมผล)
  \item Cache ลง KV
\end{enumerate}

\subsubsection{กลยุทธ์ที่ 2: JSON API (\texttt{suggest.jsp})}

HomePro และ MegaHome เปิดเผย endpoint
\texttt{/service/search/suggest.jsp?q=\{keyword\}} ที่ส่งคืนข้อมูล
สินค้าในรูปแบบ JSON structured ระบบเรียก endpoint นี้โดยตรงด้วย
HTTP GET พร้อม headers ที่จำลองเบราว์เซอร์ปกติ.

ตัวอย่างการเรียก:
\begin{lstlisting}[language=TypeScript,caption={การเรียก JSON API}]
const url = `https://www.homepro.co.th/service/search/suggest.jsp?q=${q}`;
const response = await fetch(url, {
  headers: {
    accept: "application/json",
    "user-agent": "Mozilla/5.0 ...",
    referer: "https://www.homepro.co.th/",
  },
  signal: AbortSignal.timeout(10_000),
});
const data = await response.json();
const items = data.items;  // -> [{item_name, price_display, ...}, ...]
\end{lstlisting}

\subsubsection{กลยุทธ์ที่ 3: ScrapingBee Proxy + Browser Rendering}

สำหรับเว็บไซต์ที่ block direct scraping (เช่น Thai Watsadu, BnB Home)
ระบบใช้ ScrapingBee เป็น residential proxy ที่ส่ง request ผ่าน IP
ของบ้านในประเทศไทย เพื่อหลีกเลี่ยง Cloudflare bot challenge.
หาก ScrapingBee ไม่สำเร็จ ระบบจะ fallback ไปยัง Cloudflare Browser
Rendering (\texttt{@cloudflare/puppeteer}) ที่เปิด headless Chrome
ในตัว Workers.

ลำดับ fallback ของ retail provider:
\begin{enumerate}[leftmargin=2em,itemsep=0.2em]
  \item ScrapingBee + JSON-LD canonical extraction
  \item ScrapingBee + RegEx price extraction
  \item Cloudflare Browser Rendering (Puppeteer) + DOM scraping
  \item คืน null และรอ manual upload
\end{enumerate}

\subsubsection{กลยุทธ์ที่ 4: Manual Upload}

สำหรับแหล่งที่ scrape ไม่ได้เลย (CGD anti-bot, DIT URL deprecated,
SPA ที่โหลดราคาผ่าน XHR) ระบบเปิด admin endpoint
\texttt{POST /api/admin/upload-prices} ที่รับ payload JSON หรือ CSV
จาก admin โดยตรง เพื่อบันทึกราคาเข้า KV. การ authenticate ใช้
\texttt{x-admin-token} header.

\subsection{Canonicalization (Option B)}

ปัญหาสำคัญในการเปรียบเทียบราคาข้ามแหล่งคือ \emph{ชื่อสินค้าและหน่วย
ที่ต่างกัน}. ระบบแก้ปัญหานี้ด้วยการ tag ทุก material ด้วย canonical
specification ประกอบด้วย 3 ส่วน:

\begin{itemize}[leftmargin=2em,itemsep=0.2em]
  \item \textbf{brand} — ยี่ห้อมาตรฐาน (เช่น ``ตราเสือ'', ``Q-CON'')
  \item \textbf{size} — ขนาดที่ระบุ (เช่น ``50kg'', ``DB12'',
        ``12x12 นิ้ว'')
  \item \textbf{grade} — มาตรฐาน/เกรด (เช่น ``Type I'', ``SD40'')
\end{itemize}

นอกจากนี้ ทุก material มี \texttt{searchTerms[]} ซึ่งเป็นคำค้นหาที่
หลากหลายในภาษาไทย ใช้ในการค้นจากเว็บไซต์ retail ที่ใช้คำต่างกัน.

\subsubsection{สามชั้นการกรอง (Three-tier filter)}

เมื่อระบบดึงผลลัพธ์การค้นหามาได้ จะกรองด้วยลำดับ 3 ชั้น:

\begin{enumerate}[leftmargin=2em,itemsep=0.2em]
  \item \textbf{Tight match:} ผลลัพธ์ต้องมีทั้ง \texttt{size} และ
        \texttt{brand} ตรงกับ canonical spec
  \item \textbf{Loose match:} ผลลัพธ์ต้องมี \texttt{size} ตรงเท่านั้น
  \item \textbf{Legacy match:} หากไม่มีผลลัพธ์ที่ตรง ใช้ค่ามัธยฐาน
        (median) ของผลลัพธ์ทั้งหมด — เป็น fallback ที่ใช้ในกรณีที่
        แหล่งข้อมูลไม่มีรายละเอียดเพียงพอ
\end{enumerate}

\subsection{Unit Standardization}

ระบบมี \texttt{src/lib/units/converter.ts} ที่รองรับการแปลงหน่วย
ข้ามโดเมน 12 หน่วยหลัก:

\begin{itemize}[leftmargin=2em,itemsep=0.1em]
  \item \textbf{มวล:} ตัน, กก., กรัม, ถุง 50 กก., ถุง 20 กก.,
        ถุง 1 กก.
  \item \textbf{ปริมาตร:} ลบ.ม., ลิตร, แกลลอน, ลิตรสี
  \item \textbf{ความยาว:} เมตร, ซม.
  \item \textbf{พิเศษ:} เส้น 12 ม.\ (rebar), เส้น 10 ม., ถุง 1 กก.
\end{itemize}

ตัวอย่างการแปลงราคา rebar จากเส้นเป็นตัน:
\begin{equation}
\text{ราคา/ตัน} = \frac{\text{ราคา/เส้น}}{L \times w_{pm}} \times 1000
\end{equation}
โดย $L$ = ความยาว/เส้น (ม.), $w_{pm}$ = น้ำหนักต่อเมตร (กก./ม.).

\subsection{การตรวจจับค่าผิดปกติ (Outlier Detection)}

ระบบใช้ z-score test ตามแนวทางของ \textcite{lee2024}:
\begin{equation}
z_i = \frac{|x_i - \bar{x}|}{\sigma}
\end{equation}
โดย $\bar{x}$ = ค่าเฉลี่ย, $\sigma$ = ส่วนเบี่ยงเบนมาตรฐาน.
ราคาที่มี $|z| > 2.0$ ถือเป็น outlier และจะถูก flag เป็น badge สีแดง
ในหน้า \texttt{/compare-sources} เพื่อให้ผู้ใช้พิจารณาว่าเป็นราคา
โปรโมชัน ราคาขั้นต่ำ หรือข้อผิดพลาดในการดึงข้อมูล.

ค่า threshold $z = 2.0$ เลือกเพราะให้ความสมดุลระหว่าง false positive
และ false negative สำหรับข้อมูลราคาวัสดุที่มีกระจายตัวค่อนข้างสูง
\cite{lee2024}.

\section{การคำนวณต้นทุนต่อหน่วย}
\label{sec:boq}

ระบบรองรับการคำนวณ Bill of Quantities (BOQ) สำหรับ 5 ประเภทงานหลัก
ตามหลักเกณฑ์การคำนวณราคากลางงานก่อสร้าง ตุลาคม 2560 \cite{cgd2569}:

\begin{enumerate}[leftmargin=2em,itemsep=0.2em]
  \item \texttt{wall\_tile} — งานบุกระเบื้องผนัง/พื้น
  \item \texttt{column\_beam} — งานเสาเอ็น/คานทับหลัง คอนกรีต
  \item \texttt{rebar} — งานเหล็กเสริม
  \item \texttt{paint} — งานทาสี (primer + 2 ชั้น)
  \item \texttt{brick} — งานก่ออิฐ (อิฐมอญ/อิฐมวลเบา)
\end{enumerate}

\subsection{สูตรการคำนวณตัวอย่าง: งานทาสี}

\begin{equation}
\text{ต้นทุน/ตร.ม.} = (P_{primer} \times c_{primer}) + (P_{paint} \times c_{paint} \times n)
\end{equation}
โดย:
\begin{itemize}[leftmargin=2em,itemsep=0.1em]
  \item $P_{primer}$ = ราคา primer (บาท/แกลลอน)
  \item $c_{primer}$ = consumption ของ primer (0.025 แกลลอน/ตร.ม.)
  \item $P_{paint}$ = ราคาสี (บาท/แกลลอน)
  \item $c_{paint}$ = consumption ของสี (0.031 แกลลอน/ตร.ม./ชั้น)
  \item $n$ = จำนวนชั้นที่ทา (default = 2)
\end{itemize}

\subsection{การบันทึก fact\_calculations}

ทุกครั้งที่ผู้ใช้คำนวณ BOQ สำเร็จ ระบบจะส่ง POST request ไปยัง
\texttt{/api/admin/log-calculation} เพื่อบันทึก CalcResult ลง KV
เป็น \texttt{calc:\{uuid\}} พร้อม timestamp \texttt{savedAt}.
ข้อมูลนี้สามารถใช้สำหรับการวิเคราะห์เชิงรวม เช่น สถิติการใช้งาน
ระบบ ความถี่ของวัสดุที่ถูกคำนวณ และค่าเฉลี่ยของขนาดโครงการ.

\section{การวิเคราะห์เชิงสถิติ}
\label{sec:stats}

ระบบรองรับการวิเคราะห์เชิงสถิติ 4 ประเภท เพื่อสนับสนุนการตัดสินใจ
ของผู้ประมาณราคา.

\subsection{การวิเคราะห์แนวโน้มเชิงเส้น (Linear Trend)}

ใช้สมการเชิงเส้น $Y = a + bX$ เพื่อหาแนวโน้มการเปลี่ยนแปลงของราคา
ตามเวลา \cite{ashworth2015}. ค่าสัมประสิทธิ์คำนวณด้วย Ordinary Least
Squares (OLS):

\begin{equation}
b = \frac{\sum_{i=1}^{n}(x_i - \bar{x})(y_i - \bar{y})}{\sum_{i=1}^{n}(x_i - \bar{x})^2},
\qquad a = \bar{y} - b\bar{x}
\end{equation}

ค่าสัมประสิทธิ์การกำหนด $R^2$ บ่งชี้ความสามารถของแบบจำลองในการ
อธิบายการกระจายของข้อมูล:

\begin{equation}
R^2 = \frac{\left(\sum (x_i - \bar{x})(y_i - \bar{y})\right)^2}
            {\sum (x_i - \bar{x})^2 \cdot \sum (y_i - \bar{y})^2}
\end{equation}

ค่า $R^2$ ที่เข้าใกล้ 1 หมายความว่าราคามีแนวโน้มเชิงเส้นที่ชัดเจน
ส่วนค่าที่ต่ำหมายความว่าราคามีความผันผวนเชิงไม่เป็นเส้นตรง.

\subsection{อัตราการเปลี่ยนแปลงร้อยละ (Percentage Change)}

\begin{equation}
\Delta\%_t = \frac{Y_t - Y_{t-1}}{Y_{t-1}} \times 100
\end{equation}

ใช้บ่งชี้การเปลี่ยนแปลงระหว่างช่วงเวลา (Month-over-Month, MoM) ค่าบวก
หมายถึงราคาเพิ่มขึ้น ค่าลบหมายถึงราคาลดลง.

\subsection{การวิเคราะห์ค่าเฉลี่ยและการกระจายตัว}

\begin{equation}
\text{Mean: } \bar{x} = \frac{1}{n}\sum_{i=1}^{n} x_i,
\qquad \text{Median: } \tilde{x}
\end{equation}
\begin{equation}
\text{Spread\%: } \frac{x_{max} - x_{min}}{x_{min}} \times 100
\end{equation}

ค่า Spread บ่งชี้ความแตกต่างระหว่างราคาสูงสุดและต่ำสุด หากเกิน 30\%
ระบบจะแสดงคำเตือน ``Indicative Range'' เพื่อให้ผู้ใช้ทราบว่ามีความ
ผันผวนสูงและควรพิจารณาเลือกใช้ราคาที่เหมาะสมกับบริบท.

\subsection{Regional Aggregator}

ระบบรวบรวมราคาทุกจังหวัดเป็น 6 ภูมิภาค (เหนือ/อีสาน/กลาง/ตะวันออก/
ใต้/ตะวันตก) แล้วคำนวณ \texttt{mean}, \texttt{median}, \texttt{min},
\texttt{max} ต่อภูมิภาค ผลลัพธ์ใช้สำหรับการวิเคราะห์เชิงพื้นที่และ
ใช้เป็น input ของการวางแผนต้นทุนโครงการที่ตั้งอยู่ในภูมิภาคต่าง ๆ.

\section{การ Deploy และการบำรุงรักษา}
\label{sec:deploy}

ระบบใช้ CI/CD pipeline ผ่าน GitHub Actions ที่ทำการ
\texttt{tsc --noEmit} (type-check) + build + deploy ไปยัง Cloudflare
Workers อัตโนมัติทุกครั้งที่มีการ push ไปยัง branch \texttt{main}.

นอกจากนี้ ยังมี cron schedule (\texttt{17 3 * * 1}) ที่รันทุกวันจันทร์
03:17 UTC เพื่อ:
\begin{enumerate}[leftmargin=2em,itemsep=0.1em]
  \item เรียก \texttt{POST /api/admin/refresh-prices} สำหรับทุกแหล่ง
  \item เรียก \texttt{POST /api/admin/snapshot-history} เพื่อบันทึก
        ราคาปัจจุบันเข้า history time-series
\end{enumerate}

ระบบจึงสามารถดูแลตัวเองได้โดยอัตโนมัติ ผู้ดูแลเพียงแค่ตรวจสอบ
\texttt{/health} dashboard เพื่อดู freshness ของข้อมูล และทำ manual
upload สำหรับแหล่งที่ scrape ไม่ได้เป็นรอบ ๆ.
