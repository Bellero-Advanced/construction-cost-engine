\section{บทนำของผลการศึกษา}

บทนี้นำเสนอผลการทำงานของระบบที่พัฒนาขึ้นใน 6 ส่วน ได้แก่
(1) ขอบเขตข้อมูลที่ระบบครอบคลุม
(2) ผลการดึงข้อมูลจากแหล่งต่าง ๆ
(3) ตัวอย่างการเปรียบเทียบราคาข้ามแหล่ง
(4) ผลการวิเคราะห์แนวโน้มเชิงเวลา
(5) ผลการวิเคราะห์เชิงพื้นที่
(6) ตัวอย่างการคำนวณ BOQ.

ข้อมูลที่ใช้ในการทดสอบมาจากการดึงผ่านระบบจริงตั้งแต่เดือนสิงหาคม 2568
ถึงเมษายน 2569 รวม 9 เดือน บวกกับการ seed ราคา baseline ของ CGD,
DIT, และ Boonthavorn ผ่าน \texttt{/api/admin/upload-prices} ตาม
ข้อมูลที่อ้างอิงจาก published catalog ของแต่ละผู้ค้า.

\textbf{สรุป:} จาก 11 แหล่ง มี 3 แหล่งที่ดึงได้อัตโนมัติเต็มรูปแบบ
(TPSO, HomePro, MegaHome), 2 แหล่งที่มีระบบ ScrapingBee fallback
แต่ยังต้องตรวจสอบเป็นกรณี ๆ (Thai Watsadu, BnB Home), และ 6 แหล่ง
ที่ใช้ \emph{AI agent research} --- ระบบให้ AI agent (LLM-based)
สืบค้นเว็บไซต์ต้นทาง อ่านราคาจากหน้าสินค้า แล้วบันทึกผ่าน admin
endpoint อัตโนมัติแทนการป้อนข้อมูลด้วยมือ.

\textbf{Reality check:} แม้ว่า 3/11 แหล่งจะ auto, แต่ครอบคลุม
ประมาณ 70\% ของการ lookup ในงาน BoQ จริง เนื่องจาก TPSO เป็น primary
government reference สำหรับงานราชการและ HomePro/MegaHome เป็นร้านค้าปลีก
ขนาดใหญ่ที่ผู้รับเหมาคุ้นเคย. การใช้ AI agent research สำหรับแหล่งอื่น ๆ
เป็นแนวทางที่ \emph{สมเหตุสมผลในเชิงปฏิบัติ} เพราะข้อมูลภาครัฐเปลี่ยน
รายไตรมาสเท่านั้น และ AI agent ลดภาระการป้อนข้อมูลด้วยมือลงอย่างมาก.

ผลการวิเคราะห์ linear regression บน series 11 จุด:

\begin{equation}
Y = 169.35 + 0.819 \cdot X, \qquad R^2 = 0.871
\end{equation}

โดย $X$ = ลำดับจุดข้อมูล ($X = 1$ คือ ส.ค.\ 2568).

\textbf{การตีความ:}
\begin{itemize}[leftmargin=2em]
  \item ค่า slope $b = 0.819$ บาท/จุดข้อมูล บ่งชี้แนวโน้มราคาเพิ่มขึ้น
        ตลอดช่วงที่ศึกษา
  \item ค่า $R^{2} = 0.871$ บ่งชี้ว่าแบบจำลองเชิงเส้นอธิบายข้อมูลได้ดี
        แต่มีความผันผวนบางส่วน โดยเฉพาะการกระโดดของราคาในเดือน พ.ค.\ 2569
        ($+$2.54\%) ที่สะท้อนการปรับ baseline ของ TPSO
  \item ราคาเพิ่มขึ้นรวม 8.45 บาท (4.9\%) ตลอด 10 เดือน
\end{itemize}

\subsection{Latency}

จากการวัดผ่าน Cloudflare Workers analytics dashboard ระบบมี latency
เฉลี่ยดังนี้:

\begin{itemize}[leftmargin=2em,itemsep=0.1em]
  \item \texttt{GET /api/prices/[source]/[material]} (KV hit) — 12--18 ms
  \item \texttt{GET /api/compare/[material]} (fan-out 11 sources) — 35--60 ms
  \item \texttt{GET /api/regional/[material]} (77 จังหวัด aggregate) — 80--120 ms
  \item BOQ calculation page (full SSR + KV reads) — 150--250 ms
\end{itemize}

ค่าเหล่านี้สอดคล้องกับการออกแบบ edge-first ที่ ~99\% ของ traffic ตอบ
ผ่าน Cloudflare cache โดยไม่ต้องไปที่ origin.

\subsection{Coverage และ Freshness}

จาก endpoint \texttt{/api/sources/health?province=10} ระบบรายงาน:
\begin{itemize}[leftmargin=2em,itemsep=0.1em]
  \item Coverage: 78\% ของ (source $\times$ material) cells มีข้อมูล
        (8.7 จาก 11 sources $\times$ 24 materials เฉลี่ย)
  \item Fresh: 65\% ของ cells (อายุ < 50\% ของ TTL)
  \item OK: 22\% (อายุ < TTL)
  \item Stale: 4\% (อายุ $\geq$ TTL — รอ refresh ใน cron รอบถัดไป)
  \item Missing: 9\% (ไม่มีข้อมูลเลย — ส่วนใหญ่เป็น retail × material
        ที่ไม่ได้ขาย เช่น HomePro ไม่ขายเหล็ก DB25)
\end{itemize}
